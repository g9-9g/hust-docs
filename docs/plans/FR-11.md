# Kế hoạch hoàn thiện FR-11 — Điểm thưởng & Đổi quà

## Context

PRD FR-11 (`docs/prd/04-functional-requirements.md`) yêu cầu một hệ thống điểm thưởng + đổi quà.
Hiện trạng codebase:

- **Đã có:** cộng điểm khi upvote (+2) và đạt mốc tải (+20), model `PointsTransaction`,
  API `GET /api/points/me`, trang `PointsPage` (`/me/points`).
- **Thiếu:** upload **chưa** thực sự cộng +10 điểm (enum `PointReason` chỉ có
  `UPVOTE_RECEIVED`, `DOWNLOAD_MILESTONE`, dù toast upload báo "+10 điểm"); **toàn bộ**
  phần quà tặng / đổi quà chưa tồn tại (không có model, API, trang).

Mục tiêu: hoàn thiện FR-11 ở mức MVP — cộng điểm upload, thêm catalog quà + luồng đổi quà,
trang "Đổi quà" đẹp & sống động, gộp lịch sử đổi quà vào trang điểm dưới dạng 2 tab, thêm
lối vào từ landing page.

**Quyết định phạm vi (đã chốt với người dùng):**
- **Bỏ qua admin** — không làm API/giao diện admin; quà được seed sẵn vào DB.
- **Tự động áp dụng hiệu ứng** — đổi huy hiệu / khung avatar xong là tự trang bị, hiển thị
  ngay trên Header (và khu "Bộ sưu tập" của trang đổi quà). Voucher/quà offline chỉ tạo
  bản ghi `pending`.

## Backend

### 1. Prisma schema — `server/prisma/schema.prisma`

Bổ sung enum & model:

```prisma
enum PointReason {
  UPVOTE_RECEIVED
  DOWNLOAD_MILESTONE
  UPLOAD_DOCUMENT      // mới — +10 khi upload
  REDEEM_REWARD        // mới — số âm khi đổi quà
}

enum GiftType { BADGE  AVATAR_FRAME  VOUCHER  OFFLINE_GIFT  OTHER }
enum GiftStatus { active  hidden  out_of_stock }
enum RedemptionStatus { pending  completed  cancelled }

model Gift {
  id            String     @id @default(auto()) @map("_id") @db.ObjectId
  name          String
  description   String     @default("")
  type          GiftType
  pointsCost    Int
  stock         Int?       // null = không giới hạn
  status        GiftStatus @default(active)
  // payload hiển thị hiệu ứng cosmetic
  icon          String?    // tên icon lucide cho BADGE
  accentColor   String?    // màu nhấn cho BADGE (hex)
  frameGradient String?    // css gradient cho AVATAR_FRAME
  imageUrl      String?    // ảnh cho VOUCHER/OFFLINE_GIFT/OTHER (vd /rewards/highlands-50k.png)
  createdAt     DateTime   @default(now())
  updatedAt     DateTime   @updatedAt
  redemptions   GiftRedemption[]
}

model GiftRedemption {
  id          String           @id @default(auto()) @map("_id") @db.ObjectId
  userId      String           @db.ObjectId
  giftId      String           @db.ObjectId
  giftName    String           // snapshot
  giftType    GiftType         // snapshot
  pointsSpent Int
  status      RedemptionStatus
  createdAt   DateTime         @default(now())
  user User @relation(fields: [userId], references: [id])
  gift Gift @relation(fields: [giftId], references: [id])
  @@index([userId])
}
```

- `PointsTransaction`: thêm `note String?` (để lưu "Đổi quà: <tên>").
- `User`: thêm `equippedBadgeGiftId String? @db.ObjectId`, `equippedFrameGiftId String? @db.ObjectId`,
  và relation `giftRedemptions GiftRedemption[]`.

Sau khi sửa schema chạy `npx prisma generate` (và `prisma db push` cho MongoDB).

### 2. Module quà mới — `server/src/modules/gifts/`

`gift.routes.ts` (mount `/api/gifts`):
- `GET /` — `optionalAuth`. Trả danh sách quà `status != hidden`; nếu đã đăng nhập kèm
  `balance`, danh sách `ownedGiftIds`, `equippedBadgeGiftId`, `equippedFrameGiftId`.
- `POST /:id/redeem` — `requireAuth`. Luồng chống trừ điểm 2 lần **không cần transaction
  replica-set** (Mongo standalone): dùng update có điều kiện (atomic theo document):
  1. `prisma.user.updateMany({ where: { id, contributionPoints: { gte: cost } }, data: { contributionPoints: { decrement: cost } } })` → `count === 0` thì 400 "Không đủ điểm".
  2. Nếu quà có `stock`: `prisma.gift.updateMany({ where: { id, stock: { gt: 0 } }, data: { stock: { decrement: 1 } } })` → `count === 0` thì hoàn lại điểm + 409 "Hết hàng".
  3. Tạo `GiftRedemption` (status `completed` cho BADGE/AVATAR_FRAME, `pending` cho VOUCHER/OFFLINE_GIFT).
  4. Tạo `PointsTransaction` (`REDEEM_REWARD`, `amount` âm, `note`).
  5. Nếu là cosmetic → auto-equip: cập nhật `equippedBadgeGiftId`/`equippedFrameGiftId`.
  6. Trả `{ redemption, balance, equippedBadgeGiftId, equippedFrameGiftId }`.
  - Chặn: chưa đăng nhập, quà `hidden`/`out_of_stock`, không đủ điểm.
- `POST /equip` — `requireAuth`. Body `{ giftId, slot: 'badge'|'frame' }` (hoặc `giftId: null` để gỡ).
  Xác thực user thực sự sở hữu quà đó (có `GiftRedemption`). Cho phép đổi qua lại giữa các
  cosmetic đã sở hữu ở khu "Bộ sưu tập".

`redemption.routes.ts` (mount `/api/redemptions`):
- `GET /me` — `requireAuth`, phân trang giống `points/me`. Trả danh sách `GiftRedemption`
  của user (giftName, giftType, pointsSpent, status, createdAt).

Mount routers trong `server/src/app.ts` cạnh `pointsRoutes`.

### 3. Cộng điểm upload — `server/src/modules/documents/document.routes.ts`

Trong `POST /` sau `prisma.document.create` (document mặc định `status: public`): cộng
`contributionPoints += 10` cho uploader + tạo `PointsTransaction` (`UPLOAD_DOCUMENT`,
`documentId`). Hoàn tất acceptance criteria "Điểm tăng khi upload tài liệu thành công".

### 4. Trả cosmetic trong thông tin user — `server/src/modules/auth/auth.routes.ts`

Bộ serialize user dùng cho `/me`, `login`, `register` bổ sung `equippedBadge` và
`equippedAvatarFrame` (object đã resolve từ `Gift`: name, icon, accentColor, frameGradient)
để Header render được hiệu ứng. Tạo một helper `serializeUser` dùng chung nếu chưa có.

### 5. Cập nhật `points.routes.ts`

`GET /me` trả thêm `note`; transaction `REDEEM_REWARD` có `amount` âm — không cần sửa logic
truy vấn, chỉ thêm field `note` vào response.

### 6. Seed quà — `server/src/scripts/seed.ts`

Thêm ~12 quà mẫu đủ loại, mức điểm trải từ ~50 đến ~800:
- **Cosmetic (render bằng CSS, không cần ảnh):** 3 huy hiệu (BADGE, có `icon`+`accentColor`),
  3 khung avatar (AVATAR_FRAME, có `frameGradient`).
- **Voucher / quà thật (cần ảnh):** voucher đồ uống (Highlands, Phúc Long...), Steam card,
  thẻ điện thoại, sổ tay/áo HUST... — loại `VOUCHER` (kỹ thuật số) hoặc `OFFLINE_GIFT`
  (quà vật lý), có `stock` và `imageUrl`.

**Ảnh do người dùng tự cung cấp.** Quy ước: ảnh đặt tại `client/public/rewards/`, seed trỏ
`imageUrl` tới đường dẫn `/rewards/<tên-file>`. Tôi sẽ chốt danh sách tên file (vd
`highlands-50k.png`, `steam-100k.png`, `phuclong-50k.png`...) và ghi chú rõ để người dùng
bỏ ảnh vào. Khi chưa có file, UI hiển thị placeholder (xem mục 8) nên không vỡ giao diện.

## Frontend

### 7. Types & API & hooks

- `client/src/types/index.ts`: thêm `GiftType`, `GiftStatus`, `RedemptionStatus`, `Gift`,
  `GiftRedemption`, `EquippedCosmetic`; mở rộng `User` (`equippedBadge?`, `equippedAvatarFrame?`);
  mở rộng `PointReason` (`UPLOAD_DOCUMENT`, `REDEEM_REWARD`); `PointsTransaction` thêm `note`.
- `client/src/api/gifts.ts` (mới): `getGifts()`, `redeemGift(id)`, `equipCosmetic(...)`,
  `getMyRedemptions(params)`.
- `client/src/hooks/queries.ts`: `useGiftsQuery`, `useRedeemGiftMutation`,
  `useEquipMutation`, `useMyRedemptionsQuery` — theo đúng pattern React Query hiện có
  (`keepPreviousData`, `invalidateQueries`).
- `client/src/store/auth.ts`: thêm action `patchUser`/`refreshUser` để cập nhật `balance` +
  cosmetic đang trang bị sau khi đổi quà (mutation `onSuccess` gọi vào đây).

### 8. Trang Đổi quà — `client/src/pages/RewardsPage.tsx` (mới)

Route công khai `/rewards` (KHÔNG đặt trong `ProtectedRoute`) để link từ landing page
(người chưa đăng nhập) vẫn vào xem được; nút "Đổi quà" sẽ điều hướng `/login` nếu chưa auth.

Bố cục — đẹp & sống động, **thống nhất** với hệ thống thiết kế hiện có (màu HUST burgundy,
card bo `--radius`, hover `-translate-y-1`, viền gradient, blob động):
- **Hero**: nền gradient tối + blob động (tái dùng `.animate-blob`/`.animate-blob-slow`,
  `.animate-grid-pan`), tiêu đề "Đổi quà", và **thẻ điểm** nổi bật hiển thị tổng điểm hiện
  có (icon `Award`, hiệu ứng glow). Chưa đăng nhập → CTA "Đăng nhập để đổi quà".
- **Tabs lọc theo loại** (component `Tabs` sẵn có): Tất cả / Huy hiệu / Khung avatar /
  Voucher / Khác.
- **Lưới quà**: mỗi quà là card có phần hình, tên, mô tả, badge số điểm, chỉ báo tồn kho,
  trạng thái. Hai chế độ hiển thị phần hình:
  - **Cosmetic** (BADGE/AVATAR_FRAME): preview trực tiếp bằng CSS (chip huy hiệu / vòng
    khung avatar) — không dùng ảnh.
  - **Voucher / quà thật** (VOUCHER/OFFLINE_GIFT/OTHER): hiển thị `imageUrl`. Component ảnh
    có **fallback placeholder** (card gradient màu HUST + tên quà) khi `imageUrl` rỗng hoặc
    file chưa tồn tại (`onError`) → giao diện không vỡ trước khi người dùng thêm ảnh.
  - Nút "Đổi quà" disable khi: không đủ điểm / hết hàng / ẩn / đã sở hữu (hiện "Đã sở hữu"
    hoặc "Đang dùng").
- **Modal xác nhận đổi quà** (component `dialog.tsx` mới — bọc Radix Dialog đã cài sẵn).
- **Hiệu ứng chúc mừng** khi đổi thành công: component `RewardCelebration` thuần CSS
  (sparkle/pop-in/coin-bounce) + toast `success`. Tôn trọng `prefers-reduced-motion`.
- **Khu "Bộ sưu tập của tôi"**: liệt kê cosmetic đã sở hữu, cho bật/tắt trang bị
  (gọi `useEquipMutation`).
- Link "Xem lịch sử đổi quà" → `/me/points` (tab tương ứng).
- Empty/loading state dùng `Skeleton` theo pattern hiện có.

### 9. Trang lịch sử 2 tab — `client/src/pages/PointsPage.tsx`

Giữ route `/me/points`. Giữ thẻ tổng điểm trên cùng, thêm nút CTA "Đổi quà" → `/rewards`.
Bọc nội dung trong `Tabs`:
- **Tab "Lịch sử điểm"**: danh sách `PointsTransaction` hiện tại; bổ sung label/icon cho
  `UPLOAD_DOCUMENT` và `REDEEM_REWARD`; số âm hiển thị màu đỏ, số dương màu emerald
  (hiện đang hardcode `text-emerald-600`/`+`).
- **Tab "Lịch sử đổi quà"**: dùng `useMyRedemptionsQuery` — mỗi dòng: tên quà, loại,
  điểm đã dùng, badge trạng thái (pending/completed/cancelled), thời gian. Có phân trang.

### 10. Header — `client/src/components/layout/Header.tsx`

- Thêm `NavLink` "Đổi quà" (icon `Gift`) cạnh "Tài liệu".
- Thêm mục "Đổi quà" / "Quà của tôi" trong dropdown user.
- Render **hiệu ứng cosmetic đã trang bị**: khung avatar (vòng gradient quanh `Avatar`) và
  chip huy hiệu cạnh username — tạo component nhỏ `AvatarFrame` + `BadgeChip` dùng lại được.

### 11. Landing page — `client/src/pages/LandingPage.tsx`

Thêm lối vào trang đổi quà:
- Thêm một **section "Đổi điểm lấy quà"** ngắn gọn (đặt sau "How it works"): nền nhấn,
  vài hình quà tiêu biểu, nút "Khám phá kho quà" → `/rewards`.
- Cho card feature #4 ("Điểm thưởng & quà tặng") trỏ link tới `/rewards`.

### 12. Routing & UI components

- `client/src/App.tsx`: thêm `<Route path="rewards" element={<RewardsPage />} />` (ngoài
  `ProtectedRoute`, trong `AppLayout`).
- `client/src/components/ui/dialog.tsx` (mới): bọc `@radix-ui/react-dialog` (đã có trong
  deps) theo style các component `ui` hiện có (CVA + `cn`).
- `client/src/index.css`: thêm keyframes cho hiệu ứng chúc mừng (`pop-in`, `sparkle`,
  `coin-bounce`), gói trong `@media (prefers-reduced-motion: reduce)` như các blob hiện có.

## Kiểm thử end-to-end

1. **Migrate & seed**: `npx prisma generate` + `prisma db push` trong `server/`, chạy
   `seed.ts` để có quà mẫu. Khởi động server + client.
2. **Điểm upload**: đăng nhập, upload 1 tài liệu → `+10` xuất hiện ở tab "Lịch sử điểm",
   tổng điểm tăng 10.
3. **Catalog**: vào `/rewards` (cả khi đã/đăng xuất) → thấy lưới quà, lọc theo loại chạy
   đúng. Voucher/quà thật hiện ảnh nếu đã có file trong `client/public/rewards/`, hoặc
   placeholder gradient khi chưa có — giao diện vẫn đẹp.
4. **Đổi quà — đủ điểm, cosmetic**: đổi 1 huy hiệu → modal xác nhận → thành công: điểm bị
   trừ đúng, hiệu ứng chúc mừng, huy hiệu auto-trang bị và hiện ngay trên Header.
5. **Đổi quà — không đủ điểm**: nút bị disable; gọi API trực tiếp vẫn trả 400, không trừ điểm.
6. **Hết hàng / ẩn**: quà `stock = 0` hoặc `hidden` không đổi được.
7. **Chống trừ 2 lần**: bắn 2 request `redeem` gần như đồng thời → chỉ 1 thành công, điểm
   chỉ bị trừ 1 lần (kiểm bằng `updateMany count`).
8. **Lịch sử**: `/me/points` → tab "Lịch sử điểm" có dòng `REDEEM_REWARD` âm; tab "Lịch sử
   đổi quà" liệt kê đúng giao dịch với trạng thái (voucher = `pending`).
9. **Bộ sưu tập**: sở hữu ≥2 khung avatar → bật/tắt trang bị, Header cập nhật theo.
10. **Landing**: nút "Khám phá kho quà" và card feature #4 điều hướng đúng `/rewards`.
11. Đối chiếu toàn bộ acceptance criteria FR-11 (trừ 3 mục admin — đã thống nhất bỏ qua).

## Ngoài phạm vi (đã thống nhất)

- Trang/giao diện admin và API admin quản lý quà — bỏ qua; quà quản lý qua seed.
- Thu hồi điểm khi admin xoá tài liệu vi phạm; gating điểm theo pending review (FR-16).
- Quyền "Featured document" như một loại quà — chưa đưa vào MVP này.

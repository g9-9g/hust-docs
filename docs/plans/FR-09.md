# Triển khai FR-09 (Upvote/Downvote) + cơ chế tích điểm FR-11

## Context

Dự án `hust-docs` cần hệ thống vote cộng đồng (FR-09) và cơ chế ghi nhận điểm
đóng góp (FR-11). Hiện tại:

- `Document` đã có cột `upvoteCount` / `downvoteCount` / `downloadCount` nhưng
  **chưa có model `Vote`** và **chưa có endpoint vote** — chỉ hiển thị read-only.
- Điểm hiện chỉ `+10` khi upload ([document.routes.ts:191-194](server/src/modules/documents/document.routes.ts#L191-L194)),
  chưa có lịch sử điểm (ledger).
- Frontend (`DocumentDetailPage`, `DocumentCard`) chỉ hiển thị số upvote, chưa có nút bấm.

### Quyết định sản phẩm đã chốt với chủ dự án

1. **Phạm vi lần này:** FR-09 đầy đủ + cơ chế *tích điểm* + lịch sử điểm + hiển thị
   ở profile. **Không** làm catalog quà / đổi quà / trang admin (giai đoạn sau).
2. **Upload KHÔNG cộng điểm.** Cố ý ghi đè PRD (FR-02 & FR-11 nói +10) — chỉ
   upvote và download mới sinh điểm.
3. **Điểm download = theo mốc lượt tải, đã khử trùng:** lượt tải được khử trùng để
   chống "lạm phát" (spam bấm tải). Hiển thị: đếm số *người tải khác nhau* — định
   danh bằng tài khoản (nếu đăng nhập) hoặc IP băm (nếu khách vãng lai). Điểm: mốc
   50 lượt → chủ tài liệu `+20`, **chỉ tính theo số tài khoản đăng nhập khác nhau**
   (không tính khách ẩn danh, không tính chủ tài liệu tự tải), thưởng một lần.
4. **Hủy upvote = điểm "sticky":** khi một người upvote tài liệu, chủ tài liệu
   `+2` điểm *một lần duy nhất*. Người đó bỏ upvote / đổi sang downvote / upvote
   lại — chủ tài liệu **không bị trừ và cũng không được cộng thêm**. Mỗi cặp
   (người vote, tài liệu) chỉ sinh điểm tối đa một lần.
   `upvoteCount` / `downvoteCount` vẫn phản ánh đúng trạng thái hiện tại; chỉ có
   *điểm* là sticky (đã thống nhất chấp nhận điểm có thể lệch số upvote).

### Mô hình logic vote (máy trạng thái)

Mỗi cặp `(người vote, tài liệu)` ở 1 trong 3 trạng thái `UP` / `DOWN` / `NONE`.
Quy tắc toggle (FR-09): bấm lại đúng loại vote đang có → về `NONE`; bấm loại khác
→ chuyển trạng thái. `upvoteCount`/`downvoteCount` thay đổi theo *chuyển trạng thái*.
Điểm chủ tài liệu chỉ `+2` đúng lần đầu cặp đó đạt trạng thái `UP` (cờ latch
`pointAwarded`), không bao giờ trừ lại.

## Phần 1 — Backend (`server/`)

### 1.1. Schema — `server/prisma/schema.prisma`

Thêm enum + 2 model mới, mở rộng `User` và `Document`:

```prisma
enum VoteType {
  UP
  DOWN
  NONE
}

enum PointReason {
  UPVOTE_RECEIVED
  DOWNLOAD_MILESTONE
}

model Vote {
  id           String   @id @default(auto()) @map("_id") @db.ObjectId
  userId       String   @db.ObjectId
  documentId   String   @db.ObjectId
  type         VoteType
  pointAwarded Boolean  @default(false)   // latch: cặp này đã từng sinh điểm chưa
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  user     User     @relation(fields: [userId], references: [id])
  document Document @relation(fields: [documentId], references: [id])

  @@unique([userId, documentId])          // chống vote trùng + serialize concurrency
  @@index([documentId])
}

model PointsTransaction {
  id         String      @id @default(auto()) @map("_id") @db.ObjectId
  userId     String      @db.ObjectId     // người NHẬN điểm (chủ tài liệu)
  amount     Int
  reason     PointReason
  documentId String?     @db.ObjectId
  voterId    String?     @db.ObjectId     // ai kích hoạt (với UPVOTE_RECEIVED)
  createdAt  DateTime    @default(now())

  user User @relation(fields: [userId], references: [id])

  @@index([userId])
}

model DocumentDownload {
  id         String   @id @default(auto()) @map("_id") @db.ObjectId
  documentId String   @db.ObjectId
  userId     String?  @db.ObjectId     // null = khách vãng lai
  dedupKey   String                    // "u:<userId>" hoặc "ip:<sha256(ip)>"
  createdAt  DateTime @default(now())

  document Document @relation(fields: [documentId], references: [id])

  @@unique([documentId, dedupKey])      // mỗi định danh chỉ đếm 1 lần / tài liệu
  @@index([documentId])
}
```

- `User`: thêm back-relations `votes Vote[]` và `pointsTransactions PointsTransaction[]`.
- `Document`: thêm `votes Vote[]`, `downloads DocumentDownload[]`,
  `qualifiedDownloadCount Int @default(0)` (số *tài khoản đăng nhập* khác nhau đã
  tải, không tính chủ tài liệu — dùng cho mốc điểm) và
  `downloadMilestoneAwarded Int @default(0)` (mốc cao nhất đã thưởng — chống lặp).
  `downloadCount` **đổi nghĩa** thành "số người tải khác nhau" (tài khoản hoặc IP).

Áp schema (MongoDB không dùng migration file):
```
cd server && npx prisma db push && npx prisma generate
```
Các field mới đều có default nên an toàn với dữ liệu cũ.

### 1.3. Endpoint vote — module `documents`

File mới `server/src/modules/documents/vote.routes.ts` (hoặc thêm vào
`document.routes.ts`). Đăng ký dưới `/api/documents`.

**`POST /api/documents/:id/vote`** — `requireAuth`, body `{ type: 'UP' | 'DOWN' }`
(validate bằng Zod, `z.enum(['UP','DOWN'])`).

Luồng xử lý:
1. Kiểm tra `id` hợp lệ, document tồn tại và `status: 'public'` → nếu không, 404.
2. Nếu `doc.uploaderId === req.user.id` → `400 "Không thể tự vote tài liệu của mình"`.
3. Tìm `Vote` hiện tại của `(userId, documentId)`. `current = vote?.type ?? 'NONE'`.
4. Tính `next`: nếu `current === body.type` → `'NONE'` (toggle off); ngược lại → `body.type`.
5. Tính delta count cho Document:
   - `upvoteDelta  = (next==='UP'?1:0)  - (current==='UP'?1:0)`
   - `downvoteDelta= (next==='DOWN'?1:0)- (current==='DOWN'?1:0)`
6. **Sticky points:** nếu `next === 'UP'` và `vote?.pointAwarded !== true`:
   - `prisma.user.update` → uploader `contributionPoints { increment: 2 }`.
   - Tạo `PointsTransaction { userId: uploaderId, amount: 2, reason: UPVOTE_RECEIVED, documentId, voterId }`.
   - `pointAwarded` sẽ được set `true` ở bước upsert.
7. `prisma.vote.upsert` theo khóa `userId_documentId`:
   `type: next`, `pointAwarded: vote?.pointAwarded || next==='UP'`.
8. `prisma.document.update` áp `upvoteCount`/`downvoteCount` increment (chỉ khi delta ≠ 0).
9. Trả `{ upvoteCount, downvoteCount, myVote: next==='NONE'?null:next }`.

> **Concurrency:** không dùng interactive `$transaction` (Prisma+Mongo yêu cầu
> replica set; bản dev đang chạy standalone). `@@unique([userId,documentId])`
> serialize các request trùng của cùng user; cờ `pointAwarded` đảm bảo điểm chỉ
> cộng một lần. Đủ an toàn cho MVP. *(Nếu sau này bật replica set, có thể bọc
> các bước 6-8 trong `prisma.$transaction`.)*

### 1.5. Khử trùng lượt tải + mốc điểm download

Sửa handler `GET /:id/download` ([document.routes.ts:224-234](server/src/modules/documents/document.routes.ts#L224-L234)).
Vẫn dùng `optionalAuth` (khách vãng lai vẫn tải được). Thay block
`increment downloadCount` cứng bằng luồng khử trùng:

1. **Tính `dedupKey`:** nếu có `req.user` → `"u:" + userId`; ngược lại →
   `"ip:" + sha256(req.ip)` (dùng `node:crypto`, băm để không lưu IP thô — riêng tư).
   *Lưu ý:* để `req.ip` đúng sau ngrok/Vercel, bật `app.set('trust proxy', 1)`
   trong [server/src/app.ts](server/src/app.ts) nếu chưa có.
2. **Ghi nhận lượt tải:** `prisma.documentDownload.create({ documentId, userId, dedupKey })`
   trong `try/catch`; nếu lỗi `P2002` (vi phạm `@@unique([documentId, dedupKey])`)
   → định danh này đã tải rồi, **bỏ qua đếm**, nhảy thẳng tới bước stream.
3. **Nếu là lượt tải mới (create thành công):**
   - `prisma.document.update` → `downloadCount { increment: 1 }`.
   - Nếu có `userId` **và** `userId !== existing.uploaderId` (loại tự tải):
     - `prisma.document.update` → `qualifiedDownloadCount { increment: 1 }` →
       `newQualified = existing.qualifiedDownloadCount + 1`.
     - **Mốc điểm:** khai báo `const DOWNLOAD_MILESTONES = [{ threshold: 50, points: 20 }]`.
       Với mốc cao nhất `m` thỏa `newQualified >= m.threshold` và
       `existing.downloadMilestoneAwarded < m.threshold`:
       - `prisma.user.update` → uploader `contributionPoints { increment: m.points }`.
       - `prisma.document.update` → set `downloadMilestoneAwarded = m.threshold`.
       - Tạo `PointsTransaction { userId: uploaderId, amount: m.points, reason: DOWNLOAD_MILESTONE, documentId }`.
4. Giữ nguyên phần stream file / đóng gói zip.

→ Spam bấm tải (cùng tài khoản hoặc cùng IP) không làm tăng số đếm; điểm chỉ
nhích khi đủ 50 *tài khoản đăng nhập khác nhau* tải — chống cày hiệu quả.

### 1.6. Endpoint lịch sử điểm — module `points` mới

`server/src/modules/points/points.routes.ts`, đăng ký trong `app.ts` tại `/api/points`.

**`GET /api/points/me`** — `requireAuth`. Trả:
```
{ balance: user.contributionPoints,
  transactions: [{ id, amount, reason, documentTitle?, createdAt }], ... }
```
Query `PointsTransaction` theo `userId`, `orderBy createdAt desc`, phân trang
(`page`/`limit` giống pattern list document). Join tên tài liệu qua `documentId`.

## Verification

1. **Chuẩn bị:** MongoDB chạy; `cd server && npx prisma db push && npx prisma generate`;
   chạy `npm run dev` ở cả `server/` và `client/`.
2. **Vote cơ bản:** đăng nhập user A, mở chi tiết tài liệu của user B → bấm upvote
   → `upvoteCount` +1, nút sáng, điểm của B (`/api/points/me` khi đăng nhập B) +2,
   có 1 dòng `UPVOTE_RECEIVED`.
3. **Toggle:** A bấm upvote lần nữa → `upvoteCount` -1, nút tắt; điểm của B **không đổi**.
4. **Sticky:** A upvote lại lần 3 → `upvoteCount` +1 nhưng điểm B **không cộng thêm**
   (vẫn đúng 1 transaction). A đổi sang downvote → `downvoteCount` +1,
   `upvoteCount` -1, điểm B vẫn giữ.
5. **Self-vote:** B tự vote tài liệu của mình → nhận lỗi 400.
6. **Vote state khi reload:** load lại trang chi tiết → nút vote hiển thị đúng
   `myVote` trước đó.
7. **Khử trùng tải:** cùng một tài khoản bấm tải nhiều lần → `downloadCount`
   chỉ tăng 1; khách chưa đăng nhập tải lại từ cùng IP → cũng chỉ tính 1.
8. **Mốc download:** cho 50 tài khoản *khác nhau* (không phải chủ tài liệu) tải
   → `qualifiedDownloadCount` đạt 50, chủ tài liệu `+20`, có dòng
   `DOWNLOAD_MILESTONE`; tải tiếp → không cộng nữa. Chủ tài liệu tự tải →
   không làm tăng `qualifiedDownloadCount`.
9. **Upload:** upload tài liệu mới → điểm người upload **không tăng**.
10. **Trang điểm:** mở `/me/points` → tổng điểm và lịch sử khớp các bước trên.
